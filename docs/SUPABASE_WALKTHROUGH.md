# Supabase Walkthrough

This app now uses Supabase when these environment variables exist:

```env
NEXT_PUBLIC_SUPABASE_URL=your_project_url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_anon_public_key
```

If those variables are missing, the app keeps using browser `localStorage`.

## 1. Create A Supabase Project

1. Open the Supabase dashboard.
2. Create a new project.
3. Go to Project Settings > API.
4. Copy the project URL and anon public key.
5. Create `.env.local` in the project root using `.env.local.example`.

## 2. Create Tables

Run this SQL in Supabase SQL Editor:

```sql
create table if not exists tenants (
  id bigint generated by default as identity primary key,
  name text not null,
  phone text not null default '',
  base_rent integer not null default 2200,
  previous_unit integer not null default 0,
  created_at timestamptz not null default now()
);

create table if not exists bills (
  id uuid primary key default gen_random_uuid(),
  tenant_id bigint not null references tenants(id) on delete cascade,
  tenant_name text not null,
  month text not null,
  previous_unit integer not null,
  current_unit integer not null,
  used_unit integer not null,
  unit_rate integer not null,
  electricity_amount integer not null,
  base_rent integer not null,
  total_amount integer not null,
  paid boolean not null default false,
  created_at timestamptz not null default now()
);

create index if not exists bills_tenant_id_idx on bills(tenant_id);
create index if not exists bills_month_idx on bills(month);
create index if not exists bills_created_at_idx on bills(created_at desc);
```

## 3. Create One Shared Login

Use one Supabase Auth user for both you and your brother.

1. Open Supabase Dashboard.
2. Go to Authentication > Users.
3. Click Add user.
4. Enter one shared email and password.
5. Use that same email/password on both devices.

## 4. Row Level Security For Shared Login

Because this app uses one shared account and does not have per-owner columns, allow any authenticated user to manage both tables:

```sql
alter table tenants enable row level security;
alter table bills enable row level security;

drop policy if exists "authenticated tenants access" on tenants;
drop policy if exists "authenticated bills access" on bills;

create policy "authenticated tenants access"
on tenants
for all
to authenticated
using (true)
with check (true);

create policy "authenticated bills access"
on bills
for all
to authenticated
using (true)
with check (true);
```

This is correct for one shared login. If you later want separate accounts with separate data, add an `owner_id` column and policies based on `auth.uid()`.

## 5. Development Policy

For local development without login, leave Row Level Security disabled on these two tables.

For production, do not leave private tenant data open. Add Supabase Auth and Row Level Security before sharing the app publicly.

## 6. Run The App

```powershell
npm run dev
```

On first load with empty Supabase tables, the app creates the default kiraydars automatically.

## 7. What Was Added

- `app/lib/supabase.js`: creates the Supabase browser client.
- `app/lib/rentRepository.js`: maps database snake_case rows to app camelCase objects.
- `app/page.jsx`: loads and mutates tenants/bills through Supabase when configured.
- `app/components/LoginForm.jsx`: shared email/password login screen.

## 8. Local Data Migration

The app does not automatically upload old `localStorage` data, to avoid duplicate tenants and bills.

To migrate manually:

1. Open the app before adding Supabase env vars.
2. Review the data stored locally.
3. Add Supabase env vars.
4. Recreate or import existing records using SQL or a one-time migration script.

## 9. Troubleshooting

- If you see a Supabase error, check `.env.local` and restart `npm run dev`.
- If inserts fail with an RLS error, disable RLS for development or add proper Auth policies.
- If the app falls back to browser data, check the browser alert and console message.
